CELIPA MODERNIZATION

From Monolith to Microservices

This index captures the full Celipa mobile surface area (screens), the HTTP requests each screen makes today, and how those calls map onto our target services architecture (Auth, User, Receipt, Items, Participants, OCR, Notification, Real-time, Analytics, File Storage).

Current architecture
Monolith

Celipa today uses a single Node/Express server (celipa-server) with MongoDB, where all concerns (auth, users, receipts, items, participants, OCR orchestration, payments, notifications, dashboard) live behind the same base URL https://server.celipa.co. 

// Example (current) mobile HTTP calls
POST /user/login
POST /user/signup
PATCH /user/profile
POST /receipt/create
GET  /user/receipt/:userId
GET  /user/menuitems/:receiptId
PUT  /menu/addparticipant/:userId/:menuId/:receiptId/:quantity
PUT  /receipt/confirmStatus/:userId/:receiptId
GET  /global/credentials        // Veryfi OCR
POST (Veryfi) /partner/documents/

All of these share a single DB and Socket.io instance; scaling one concern forces scaling the whole server.

Target architecture
Target services

Celipa is refactored into domain-aligned target services (Auth, User, Receipt, Items, Participants, OCR, Notification, Real-time, Analytics, File Storage) behind an API Gateway. Each service owns its data and surface. 

// Example (target) service endpoints
POST /auth/login                     → Auth Service
PATCH /users/:userId                 → User Service
POST /receipts                       → Receipt Service
GET  /receipts/:id                   → Receipt Service
GET  /receipts/:id/items             → Receipt Items Service
POST /receipts/:id/items/:itemId/assign
                                     → Receipt Items Service
POST /receipts/:id/participants      → Participants Service
PUT  /receipts/:id/participants/:userId/confirm
                                     → Participants Service
POST /ocr/scan                       → OCR Service
POST /notifications/email            → Notification Service
WS   wss://realtime.celipa.co       → Real-time Service

The mobile app keeps the same flows; only the backing URLs and service boundaries change.

Target services (what we can reuse)

Celipa's target architecture reuses a few existing services from the broader platform and introduces several Celipa-specific services. This section summarizes the main reusable services and what new services we still need to build.

Reusable target services
Existing services
Target service Source Celipa usage
IAM Service dl/iam/v1
(auth_code, user, groups, organizations, refresh tokens) 		Use as the foundation for Celipa authentication and authorization: user identities, sessions, group/organization membership and tokens. Celipa's Auth and User services become thin adapters over the IAM APIs instead of re‑implementing identity.
Email Service dl/messaging/v1, dl/sg/mail/v1, dl/sendgrid/v1
(generic SendGrid integration) 		Reuse as the email channel for Celipa's Notification Service: signup/verification emails, receipt summaries, incident emails and marketing. We will extend this with Twilio SMS to cover the text flows Celipa already uses today.
New Celipa-specific services
To be built

The remainder of Celipa's backend remains Celipa-specific and will be implemented as new target services, even when conceptually similar to existing platform services:

  • Receipt Service – create/read/update receipts, generate receipt codes, expose /receipts and /receipts/:id.
  • Receipt Items Service – menu items and assignment APIs, e.g. /receipts/:id/items, /receipts/:id/items/:itemId/assign.
  • Receipt Participants Service – participants, join codes, payment status, e.g. /receipts/:id/participants.
  • OCR Service – wraps Veryfi/AWS Textract for Celipa’s receipt scanning needs.
  • Notification Service – composes the reusable email service with Twilio SMS and (optionally) push notifications for Celipa-specific templates.
  • Real-time Service – Socket.io/WebSocket hub for receipt rooms and payment updates.
  • Analytics Service – aggregates user and receipt statistics for the mobile home dashboard and any future admin dashboards.
  • File Storage Service – S3-backed storage for receipt images and profile pictures.

Optionally, the existing dl/gateway/v1 service can be reused as an API gateway, but this is not required: Celipa can also be fronted by a dedicated gateway that simply routes to the IAM, messaging and the new Celipa-specific target services.

End-to-end flows (monolith → microservices)

Receipt creation flow
Scan → OCR → Create 
HomeScreen
  └─ tap "Scan receipt"
       ▼
ReceiptPictureScreen
  ├─ GET  /global/credentials             → OCR creds
  ├─ POST Veryfi /partner/documents/      → OCR extraction
  └─ navigate(UserReceiptScreen, data)
       ▼
UserReceiptScreen
  └─ POST /receipt/create                 → create receipt + items + participants
       ▼
VirtualReceiptScreen
  ├─ GET  /user/menuitems/:receiptId
  ├─ GET  /current/user/:receiptId
  ├─ WS   Socket.io on <receiptId>
  └─ collaborative splitting begins

In the microservices world, these map to OCR, Receipt, Items, Participants, and Real-time services, but the app flow stays identical.

Join & pay flow
Join → Select → Pay 
HomeScreen
  └─ tap "Join receipt"
       ▼
JoinReceiptScreen
  ├─ GET /receipt/join/:userId/:code
  └─ navigate(VirtualReceiptScreen, { receiptId })
       ▼
VirtualReceiptScreen
  ├─ assign items (PUT /menu/addparticipant/...)
  └─ tap "Confirm selection"
       ▼
AcceptPaymentModal
  └─ PUT /receipt/confirmStatus/:userId/:receiptId
       ▼
ConfirmationScreen
  └─ done

With microservices, the same flow uses Receipt, Participants, Calculation, Notification, and Real-time services instead of a single Express app.

Real-time & events

Socket.io channels
Real-time Service

Celipa already uses Socket.io heavily in VirtualReceiptScreen and PaymentUpdatesScreen. The microservices migration keeps the same client code, but points the connection at a dedicated Real-time service rather than the monolith. 

// Current
const SocketEndpoint = SERVER_URL + "/";
const socket = io(SocketEndpoint, { transports: ["websocket"] });

socket.on(receiptId, info => {
  // receipt, menu, participants, tip/tax
});

socket.on(`${receiptId}pu`, info => {
  // payment updates
});
// Target
const REALTIME_URL = "wss://realtime.celipa.co";
const socket = io(REALTIME_URL, { transports: ["websocket"] });

// Event payloads emitted by Real-time Service in response
// to domain events on the message bus:
//
// - receipt.updated
// - item.assigned
// - participant.added
// - tip.updated
// - payment.confirmed

This mirrors the target services pattern where each domain emits events into a shared bus and presentation/frontends subscribe via a gateway and real-time layer.
